CostVision · Should-Cost Engine

How Auto SW Cost is built

A bottom-up parametric model that decomposes a premium vehicle's software into 49 costed modules, prices each from engineering first principles, then reports its own error against real programmes.

49 modules 7 domains ISO 26262 / 21434 Monte-Carlo P10 / P50 / P90 back-test MAPE 24.9% GBP throughout

01The pipeline

Every estimate flows through the same five stages. Nothing is a single guessed number — the total is assembled from 49 independent module estimates and then stress-tested.

STAGE 1

Decompose

Programme → 49 modules across 7 domains, each with its own cost DNA.

STAGE 2

Cost each

Per-module formula: effort × sourced rate, split into dev / test / lifecycle.

STAGE 3

Roll up

Sum to programme total, NRE and £/vehicle, grouped by domain.

STAGE 4

Simulate

1,000-run Monte-Carlo, ρ=0.55 correlated → P10 / P50 / P90 band.

STAGE 5

Validate

Back-test vs 7 published programmes; report the model's own variance.

03The per-module formula

Each module builds a fully-loaded labour rate, converts base effort through reuse and ASIL, then splits development into engineering sub-buckets. Testing scales with ASIL (an ASIL-D module verifies harder than it develops), and multi-year costs are added on top.

rate = ukBase × region × devSource × seniority × overhead × schedule
effortPM = basePersonMonths × reuse  →  devPM = effortPM × asildev
dev = reqs 12% + arch 14% + algo 22%·cx + impl 37%·cx' + safety 15%·floor
test = dev × testFrac × (asiltest ÷ ref)  // SIL·MIL·HIL·regression·pentest
+ integration · cybersecurity · calibration  (NRE)
+ (maintenance + cloud + IP licence) × NPV factor  (lifecycle)

Reuse can't erase safety. The safety-compliance slice is floored by ASIL — reused ASIL-C/D code still needs re-verification and re-qualification.

Correlated uncertainty. Overruns move together: a slipped schedule inflates dev, test and integration at once. The Monte-Carlo blends a shared draw at ρ=0.55, so the tail isn't understated.

09So — how accurate is it?

An envelope, not audited-to-the-pound. This is a ±25–35% parametric should-cost model. The validation targets are themselves third-party analyst estimates, not internal books — so it validates a range, and its real value is that it tells you its own confidence band rather than pretending to a precision the data doesn't support.

Costs by domain, not by brand. It captures the cost of Burmester audio, E-Active Body Control or Digital Key — but won't enumerate every marketing-named ECU. That would be false precision.

Only as good as its inputs. Give it the right region, source, volume, reuse and ASIL and it lands in-band. It's built to be driven by a cost engineer — every rate overrideable with your data.

Strong enough to challenge a supplier quote or set a target cost; honest enough to show its error bars while doing it.
